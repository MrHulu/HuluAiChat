### 1.2 Sidecar Health Monitoring - Proposed Solution

**Problem**: Rust compilation fails due to memory constraints.

**Recommended Approach**: Frontend-based health monitoring (bypasses Rust compilation entirely)

```
Architecture:
+-----------------+        HTTP GET        +------------------+
|   Frontend      | ----> /api/health ---- |  FastAPI Sidecar |
|   (React)       |        every 5s        |  (Python)        |
+-----------------+                        +------------------+
        |
        +-- On 3 consecutive failures --> Show "Backend Offline" banner
        +-- On recovery --> Hide banner, reconnect WebSocket
```

**Implementation**:
1. Create `useBackendHealth` hook - polls `/api/health` every 5 seconds
2. Create `BackendOfflineBanner` component - shows restart button
3. Use Tauri shell plugin to restart sidecar process
4. Store failure count in component state (not persistent)

**Complexity**: Low (1-2 days)
**Impact**: High (improves resilience, user experience)

### 1.3 Global Service Instance Pattern

**Current State**: Several services use global singleton pattern:
- `mcp_service = MCPService()` in `mcp_service.py`
- This pattern reduces testability and makes dependency injection harder

**Recommendation**: Do NOT refactor now. The pattern works for a single-developer desktop app. Refactoring to dependency injection adds complexity without immediate benefit.

**Rationale**:
- "If it ain't broke, don't fix it"
- No production issues from current pattern
- Testing is functional with mock patches
- Complexity budget is better spent on features

## 2. Performance Optimization Opportunities

### 2.1 Startup Time Analysis

**Current Bottlenecks**:
1. Backend sidecar spawn: ~2-3 seconds
2. Frontend initial render: ~500ms
3. ChromaDB lazy load: On-demand (acceptable)

**Optimization Recommendations**:

| Area | Current | Target | Method | Priority |
|------|---------|--------|--------|----------|
| Backend startup | ~2.5s | ~1.5s | Defer ChromaDB init to first use | P2 |
| Frontend bundle | ~1.8MB | ~1.2MB | Code split Mermaid/KaTeX | P3 |
| Initial render | ~500ms | ~300ms | Defer non-critical components | P3 |

**Recommendation for v3.58.0**: Focus on backend startup optimization only. Code splitting is lower priority as current bundle size is acceptable for a desktop app.

### 2.2 Memory Footprint

**Current**: ~250MB typical
**Target**: < 200MB

**Strategies**:
1. **Virtual list enforcement**: Already using `@tanstack/react-virtual` - verify it's active for all long lists
2. **Message cache limits**: Keep only last 100 messages per session in memory (lazy load history on scroll up)
3. **ChromaDB memory**: Already lazy-loaded - no changes needed

**Recommendation**: Implement message cache limit in v3.58.0 if time permits.

### 2.3 Response Latency

**Current bottlenecks**:
1. First API call after idle (backend "cold" state)
2. WebSocket chunk rendering (UI jank on fast streams)

**Solutions**:
1. **Backend warmup**: Already have `/api/health` ping on startup
2. **Chunk batching**: Use `requestAnimationFrame` for stream rendering in ChatView

**Recommendation**: Chunk batching is a P2 optimization. Current streaming UX is acceptable.

### 3.2 What NOT to Do

Following the principle of avoiding unnecessary complexity:

1. **NO microservices split** - Desktop app doesn't need it
2. **NO message queue** - Overkill for local app
3. **NO cloud sync** - Violates privacy-first principle (unless Boss explicitly requests)
4. **NO GraphQL** - REST is sufficient for current API surface
5. **NO framework migration** - Tauri 2.0 + React 19 is modern and stable
