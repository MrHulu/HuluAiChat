**审核者**: Critic Munger (Charlie Munger 思维模型)
**审核日期**: 2026-03-12
**审核对象**: `docs/v3.58.0-roadmap.md`

## 一句话判断

**需要修改** - 方向基本正确，但存在功能重复、优先级错配和隐含的复杂度风险。

---

## 一、致命问题（必须修复）

### 1.1 TASK-201 消息复制按钮 - 功能已存在

**问题**: 规划中 TASK-201 要做"消息复制按钮"，但这个功能**已经实现**。

**证据**: 在 `MessageItem.tsx` 第 467-492 行：
```tsx
// Copy button for all messages
{!isEditing && !isStreaming && (
  <button onClick={handleCopy} ...>
    {isCopied ? <Check /> : <Copy />}
  </button>
)}
```

**这会如何杀死我们**:
- 浪费 0.5 Cycle 开发已存在的功能
- 让团队看起来不知道自己有什么功能
- 浪费测试资源

**建议**: 立即删除 TASK-201，或重新定义为"增强复制功能"（如复制 Markdown 源码、复制带格式等）

---

### 1.2 TASK-200 消息引用回复 - 部分功能已存在

**问题**: MessageItem 已经有 `onQuote` prop 和引用按钮（第 562-581 行），双击引用也已实现（第 339-344 行）。

**现有代码**:
```tsx
// Quote/Reply button for all messages - Cycle #145
{onQuote && !isEditing && !isStreaming && (
  <button onClick={...onQuote(message)} ...>
    <Quote />
  </button>
)}
```

**缺失部分**:
- 输入框的引用预览组件
- 后端 API 支持 `quoted_message_id`

**这会如何杀死我们**:
- 重复造轮子
- 预估 1 Cycle 实际只需要 0.5 Cycle

**建议**: 重新定义 TASK-200 为"完善引用回复功能"，明确只需：
1. 前端：QuotePreview 组件
2. 后端：Chat API 支持 quoted_message_id

预计周期应从 1 Cycle 改为 0.5 Cycle。

## 二、重要问题（应该修复）

### 2.1 主题定制复杂度被低估

**问题**: 规划说"1 Cycle"，但实际复杂度更高。

**现状分析**:
- 已有 `ThemeProvider` 只支持 `dark | light | system` 三种预设
- TASK-203 要求：
  - 预设主题 (已有基础)
  - **自定义颜色** (主色、背景色、强调色) - 这是新功能
  - 实时预览
  - CSS 变量动态切换

**复杂度来源**:
1. Tailwind v4 的 CSS 变量系统需要重新设计
2. 自定义颜色需要动态生成完整的颜色调色板（不是换个 hex 值那么简单）
3. 对比度、可访问性检查
4. 深色/浅色两套自定义主题的协调

**这会如何杀死我们**:
- 1 Cycle 变成 3 Cycle
- 用户体验差（颜色搭配不协调）
- 无障碍问题（对比度不足）

**建议**:
- MVP 只做预设主题选择（已有基础的完善）
- 自定义颜色作为 P2 功能，单独规划
- 或者明确限制为"选择主色调，自动生成配色"

---

### 2.2 快捷键自定义与现有系统冲突

**问题**: 规划说"无需后端支持"，但快捷键自定义会与多个现有功能产生交互。

**潜在冲突**:
1. 现有 `CommandPalette` 有硬编码快捷键
2. `MessageItem` 有双击引用
3. 系统级快捷键（Ctrl+C 复制等）
4. 跨平台差异（macOS Cmd vs Windows Ctrl）

**这会如何杀死我们**:
- 快捷键冲突导致功能失效
- 用户体验混乱
- 测试复杂度指数增长

**建议**:
- 先梳理现有所有快捷键
- 明确哪些可配置、哪些不可配置
- 增加冲突检测的复杂度预估

---

### 2.3 会话内搜索与全局搜索混淆

**问题**: 用户可能混淆"会话内搜索"和现有的"会话搜索"。

**现有功能**:
- 已有会话列表搜索（搜索会话标题）
- 已有帮助文档搜索

**新功能**:
- TASK-202 会话内搜索（搜索消息内容）

**这会如何杀死我们**:
- 用户不知道 Ctrl+F 是搜索什么
- 搜索入口不统一
- 交互混乱

**建议**: 明确交互设计：
- Ctrl+F = 会话内搜索（当前会话消息）
- Ctrl+K → 搜索会话 = 全局搜索
- 或者提供一个统一搜索入口

---

## 三、建议改进（可以考虑）

### 3.1 P2 功能（PDF/HTML 导出）价值存疑

**逆向思考**: 为什么用户需要导出？

**可能场景**:
1. 备份 - 但已有数据库备份
2. 分享 - 但规划说"不涉及服务器"
3. 存档 - 可能有用

**问题**:
- PDF 导出涉及复杂的样式渲染
- 代码高亮、Markdown、数学公式在 PDF 中的呈现
- 跨平台字体问题

**建议**:
- MVP 先只做纯文本/Markdown 导出
- HTML 导出比 PDF 简单，可以作为 MVP
- PDF 放到 P2 或 v3.59.0

---

### 3.2 缺失的重要功能

**逆向思考**: 如果这个版本失败了，可能是因为缺少什么？

**被忽略的高价值功能**:
1. **消息搜索历史** - 跨会话搜索（比单会话搜索价值更高）
2. **会话导出导入** - 迁移、备份场景
3. **快捷命令** - "/" 触发的快捷输入

**建议**: 评估是否有一个应该放入 MVP。

---

### 3.3 周期预估过于乐观

**当前预估**: 4-5 Cycles (MVP)

**我的预估**:
- TASK-200: 0.5 Cycle (已有基础)
- TASK-201: 0 Cycle (已存在)
- TASK-202: 0.5-1 Cycle (简单但需要 UX 设计)
- TASK-203: 1-2 Cycle (复杂度被低估)
- TASK-204: 1-2 Cycle (冲突处理复杂)
- Phase 3: 1 Cycle

**总计**: 4-6.5 Cycles，更可能是 5-6 Cycles

## 四、Pre-Mortem 分析

假设 v3.58.0 发布后**失败了**，最可能的原因：

### 场景 1: 功能重复导致负面评价
> "他们居然把复制按钮当做新功能发布？"
>
> **根因**: 不知道自己有什么功能
>
> **预防**: 删除 TASK-201，审核 TASK-200

### 场景 2: 主题定制太难用
> "自定义颜色后看起来像个调色盘爆炸了。"
>
> **根因**: 颜色生成算法不成熟，缺乏设计约束
>
> **预防**: 限制自定义范围，或只提供预设

### 场景 3: 快捷键冲突
> "我改了快捷键，结果什么都按不出来了。"
>
> **根因**: 冲突检测不完善
>
> **预防**: 禁止配置关键系统快捷键

### 场景 4: 搜索功能混乱
> "我搜索了，但没找到我要的东西。"
>
> **根因**: 多个搜索入口不清晰
>
> **预防**: 统一搜索 UX 或明确区分
